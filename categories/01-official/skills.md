## Problem & Solution

### Core Problems

The openclaw/skills repository addresses the problem of **persistent backup and historical archiving for the skill registry**:

1. **Data durability**: ClawHub, as an online service, needs a reliable backup mechanism to prevent data loss
2. **Historical traceability**: All versions of all skills need to be preserved, including deleted or flagged-as-malicious skills, for security analysis and auditing
3. **Offline access**: Developers and researchers need offline access to the skill library for batch analysis or research
4. **Transparency**: A public Git repository provides fully transparent skill publishing history that anyone can review
5. **Disaster recovery**: If the ClawHub service experiences an outage, all data can be recovered from the GitHub repository

### Solution

The openclaw/skills repository serves as the **official backup archive for ClawHub**, with the following characteristics:

- **Automated backup**: Through GitHub App integration, every skill publish/update on ClawHub is automatically synced to this repository
- **Complete version history**: All versions of every skill are preserved, including soft-deleted versions
- **Structured storage**:
  - One directory per skill (named by username)
  - One subdirectory per version (named by version number or identifier)
  - Contains `SKILL.md` and `_meta.json` (version metadata)
- **Security analysis corpus**: Retains suspicious/malicious skills for further analysis, helping improve security detection algorithms
- **Read-only archive**: Users should preferentially use the ClawHub website or CLI to download skills; this repository serves only as a historical archive

### Directory Naming Convention

- **First-level directories**: Username (GitHub handle)
  - e.g., `00xmorty`, `steipete`, `anthropic`
- **Second-level directories**: Skill slug or version identifier
  - e.g., `conatus`, `my-skill-v1.0.0`
- **Files**:
  - `SKILL.md`: The skill's main document (required)
  - `_meta.json`: Version metadata (optional, generated by the backup system)
  - Other supporting files (scripts, configs, examples, etc.)

### Backup Mechanism

Based on ClawHub's documentation (`docs/github-import.md`, `githubBackups.ts`), the backup workflow is as follows:

1. **Trigger conditions**:
   - A user publishes a new skill version on ClawHub
   - A user updates an existing skill
   - An admin manually triggers a backup

2. **GitHub App integration**:
   - ClawHub authenticates via a GitHub App
   - Has write permissions to the `openclaw/skills` repository
   - Creates/updates files via the GitHub API

3. **Backup content**:
   - All text files of the skill (downloaded from Convex `_storage`)
   - Version metadata (`_meta.json`):
     - Version number, publish time, changelog
     - Author information, tags
     - File list and hash values

4. **Conflict handling**:
   - If the target path already exists, a version suffix is used to avoid overwriting
   - All historical versions are preserved

5. **Malicious skill handling**:
   - Skills flagged as malicious or soft-deleted are still retained in the repository
   - Used for security research and improving detection algorithms
   - README explicitly warns users to exercise caution

```python
# apps/api/services/git_restore.py
class GitRestoreService:
    async def restore_agent_config(
        self,
        agent_id: str,
        version: str,
        db: AsyncSession
    ):
        """Restore agent configuration from GitHub"""
        path = f"agents/{agent_id}/{version}/config.yaml"

        try:
            file_content = self.repo.get_contents(path)
            config = yaml.safe_load(file_content.decoded_content)

            # Restore to database
            agent = await db.get(Agent, agent_id)
            agent.config = config
            agent.version = version
            await db.commit()

            return agent
        except GithubException as e:
            if e.status == 404:
                raise ValueError(f"Config not found: {path}")
            raise
```
